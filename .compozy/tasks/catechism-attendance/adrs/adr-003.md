# ADR-003: Estratégia Offline — PWA com IndexedDB + Background Sync

## Status

Accepted

## Date

2026-05-04

## Context

Catequistas precisam marcar presença no celular durante a aula, mas a paróquia tem sinal de internet instável. O registro de presença não pode ser perdido. A solução deve funcionar no navegador do celular sem instalação via loja de aplicativos (requisito do PRD).

## Decision

Implementar **PWA (Progressive Web App)** com **Service Worker** para cache de assets e **Dexie.js (IndexedDB)** para armazenamento local das chamadas pendentes. Ao recuperar conexão, o Service Worker dispara sincronização automática via **Background Sync API** (com fallback de polling no `online` event). O endpoint `POST /api/attendance` é idempotente por `session_id` para prevenir duplicatas no reenvio.

## Alternatives Considered

### Alternative 1: Armazenar em localStorage

- **Description**: Salvar chamadas pendentes em `localStorage` como JSON.
- **Pros**: API simples, sem dependências externas.
- **Cons**: Limite de 5–10 MB por origem; síncrono (bloqueia thread); sem suporte a queries; não adequado para dados estruturados.
- **Why rejected**: IndexedDB é superior para dados estruturados e não tem o limite restritivo do localStorage.

### Alternative 2: App nativo com SQLite offline-first

- **Description**: React Native com SQLite local (ex: WatermelonDB) e sync com o servidor.
- **Pros**: Offline robusto e nativo; SQLite é mais poderoso que IndexedDB.
- **Cons**: Requer distribuição via app store; contradiz o requisito de "funcionar no browser sem instalação".
- **Why rejected**: Fora do escopo definido no PRD.

## Consequences

### Positive

- Catequista pode fazer a chamada completa sem internet; dados sincronizam automaticamente.
- Indicador visual claro (banner "modo offline") informa o catequista sobre o status.
- Idempotência no endpoint elimina risco de chamadas duplicadas no reenvio.
- PWA pode ser "instalado" na tela inicial do celular sem app store.

### Negative

- Background Sync API não tem suporte no Safari/iOS (limitado a Chrome/Android). Fallback necessário: listener no evento `online` do browser.
- IndexedDB tem API verbosa; Dexie.js abstrai isso mas adiciona dependência.

### Risks

- iOS Safari não suporta Background Sync: mitigado com fallback de `window.addEventListener('online', sync)`.
- Dados no IndexedDB são por origem e podem ser limpos pelo browser em pressão de storage: mitigado com indicador de pendências e confirmação ao catequista.

## Implementation Notes

- Usar `next-pwa` ou configuração manual de Service Worker com `workbox`.
- Schema Dexie: tabela `pending_sessions` com campos `id`, `classId`, `date`, `catechistId`, `records[]`, `createdAt`.
- Sync endpoint: `POST /api/attendance` aceita array de sessões; usa `upsert` no Supabase com `ON CONFLICT DO NOTHING` por `session_id`.
- Testar offline em Chrome DevTools (Network → Offline) e em dispositivo Android real.
